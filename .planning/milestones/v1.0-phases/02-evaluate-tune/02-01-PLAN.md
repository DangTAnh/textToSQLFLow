  <action>
    Create `text_to_sql_flow/evaluator.py` with the following components:

    **1. EvaluationResult Pydantic model:**
    ```python
    class EvaluationResult(BaseModel):
        score: float          # Overall score 0-10
        feedback: str         # Detailed feedback text from LLM
        dimensions: dict[str, float]  # Per-dimension scores
        passed: bool          # score >= THRESHOLD
    ```

    **2. Constants:**
    - `THRESHOLD = 7.0` (per EVAL-02)
    - `MAX_ITERATIONS = 5` (per EVAL-04 — used by pipeline in Plan 02, defined here as single source of truth)
    - `EVALUATOR_MODEL = "gpt-4o"` (same as generation model)

    **3. EVALUATOR_SYSTEM_PROMPT** (string constant):
    A detailed rubric instructing the LLM to evaluate a Spark SQL ETL flow JSON on 5 dimensions, each scored 1-10:

    - **correctness**: SQL syntax correctness, proper Spark SQL functions, no logical errors
    - **completeness**: All required steps from the business description are covered, intermediate transformations exist
    - **spark_best_practices**: Uses broadcast hints for small tables, avoids UDFs where built-in functions suffice, proper partitioning, uses temp views appropriately
    - **dependency_correctness**: Step parents list matches actual table/view dependencies, execution order is correct (parallel steps have same order number)
    - **code_quality**: SQL is readable, uses consistent naming (UPPER_SNAKE_CASE), descriptions are meaningful, diagram positions are reasonable

    Output format instruction: respond with ONLY valid JSON in this shape (no extra text):
    ```json
    {
        "score": 7.5,
        "dimensions": {
            "correctness": 8,
            "completeness": 7,
            "spark_best_practices": 6,
            "dependency_correctness": 8,
            "code_quality": 7
        },
        "feedback": "The flow covers the main requirements but... [detailed feedback]"
    }
    ```

    **4. `build_evaluation_prompt(flow_dict: dict) -> tuple[str, str]`:**
    - Returns (EVALUATOR_SYSTEM_PROMPT, user_prompt)
    - user_prompt wraps the flow JSON as a code block: `Evaluate the following Spark SQL ETL flow:\n\n\`\`\`json\n{json.dumps(flow_dict, indent=2)}\n\`\`\``

    **5. `parse_evaluation_response(response_text: str) -> EvaluationResult`:**
    - Reuses the same JSON extraction pattern from `flow_parser.py`: strip markdown code blocks, find first { and last }, parse with json.loads
    - Validates the parsed dict has required keys (score, dimensions, feedback)
    - Computes `passed` = score >= THRESHOLD
    - Raises ValueError if JSON is missing or malformed (caller retains)
    - Falls back gracefully: if dimensions dict is missing from response, set to empty dict (not fail)

    **6. `evaluate_flow(flow_path: Path) -> EvaluationResult`:**
    - Reads the flow JSON file from `flow_path`
    - Calls `build_evaluation_prompt()` with the parsed dict
    - Calls `call_llm()` with the prompt pair (reuses existing client)
    - Calls `parse_evaluation_response()` to extract result
    - Uses the same `call_llm` function from `text_to_sql_flow.llm.client` — no retry wrapper here since `call_llm` already retries internally

    Import pattern:
    ```python
    import json
    import logging
    from pathlib import Path
    from pydantic import BaseModel
    from text_to_sql_flow.llm.client import call_llm
    ```

    Use the same JSON extraction logic as `flow_parser.py` (regex for ```json blocks, find first { / last }) so the evaluator handles the same LLM response variability.

    Log warning and raise ValueError if the response cannot be parsed after the LLM call succeeds (the caller in pipeline.py will handle it).
